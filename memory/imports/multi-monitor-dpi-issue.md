# Проблема: overlay не заполняет экран в мульти-мониторной конфигурации

## Симптомы

- При клике на иконку в трее создаётся по одному overlay-окну на каждый монитор.
- Overlay отображается не на весь экран: изображение/масштаб выглядят так, будто окно занимает только часть монитора.
- Пользователь сообщает, что "захваты получились для каждого монитора не на весь экран".

## Конфигурация

На момент диагностики обнаружены два монитора:

| Монитор | Имя устройства | Bounds (физические пиксели) | Scaling |
|---------|----------------|----------------------------|---------|
| 1       | `\\.\DISPLAY1` | `(-2560, 778, 2560, 1600)` | 1.75    |
| 2       | `\\.\DISPLAY5` | `(0, 0, 3840, 2160)`       | 2.00    |

## Что уже сделано

### 1. Отдельный захват на каждый монитор

Раньше использовался один общий захват всего виртуального рабочего стола с последующим кропом. Это приводило к смешению физических и логических координат при разном DPI.

Теперь для каждого монитора делается свой захват:

```csharp
var capturedImage = await captureService.CaptureScreenAsync(screen);
```

### 2. Получение физических координат мониторов

`GetScreens()` переписан с использованием `EnumDisplayDevices` + `EnumDisplaySettings`. Это возвращает физические позиции и разрешения мониторов (`dmPositionX/Y`, `dmPelsWidth/Height`), а не логические координаты из `GetMonitorInfo`.

### 3. Захват через DC конкретного монитора

`CaptureScreen()` пытается создать DC устройства через `CreateDC(screen.Name, ...)` и захватить область размером с `GetDeviceCaps(HORZRES/VERTRES)`. Если не получается — fallback на `CaptureRegion()`.

### 4. Позиционирование и размер overlay-окна

В `CaptureOverlayWindow.StartCapture()`:
- окно показывается через `Show()`;
- затем через `Dispatcher.UIThread.Post(..., DispatcherPriority.Loaded)` вызывается `SetWindowPos` с физическими координатами и размером монитора;
- размеры окна в DIP вычисляются как `screenBounds / scaling`.

### 5. Диагностика

- В `debug.log` пишутся обнаруженные мониторы, scaling, размеры захваченных изображений.
- Каждый захват сохраняется в `%LOCALAPPDATA%\Histshot\capture_{N}_{DeviceName}_{width}x{height}.png`.

## Текущее состояние

Захваченные изображения имеют правильные физические размеры:

- `capture_0_DISPLAY1_2560x1600.png` — 2560×1600 пикселей;
- `capture_1_DISPLAY5_3840x2160.png` — 3840×2160 пикселей.

Это означает, что **захват экрана работает корректно**. Проблема остаётся на этапе **отображения overlay-окна**: окно не заполняет весь монитор.

## Возможные причины

1. **Окно открывается не на том мониторе.**
   - Если overlay для `DISPLAY1` открывается на `DISPLAY5`, его DPI scaling будет 2.0 вместо 1.75, и физический размер окна станет меньше/больше нужного.

2. **`RenderScaling` окна не соответствует монитору.**
   - В предыдущих версиях `GetDpiForWindow()` мог возвращать DPI основного монитора, если окно ещё не было перемещено.
   - Сейчас используется scaling из `GetDpiForMonitor()` для конкретного монитора.

3. **Avalonia не применяет `Position` до `Show()` корректно для отрицательных/внеprimary координат.**
   - `DISPLAY1` имеет `X = -2560`, то есть расположен слева от основного монитора.
   - Возможно, окно создаётся на primary мониторе, а только потом перемещается.

4. **`SetWindowPos` не перемещает окно на другой монитор до конца.**
   - Хотя `SetWindowPos` перемещает HWND, Avalonia может не сразу обновить `RenderScaling` или `Bounds`.

5. **Размеры `Width`/`Height` в DIP не совпадают с физическим размером после `SetWindowPos`.**
   - Если `SetWindowPos` установил физический размер, а Avalonia затем переопределила `Width`/`Height` по своему scaling, окно может сжаться/растянуться.

## Что нужно проверить

1. Запустить приложение, кликнуть по трею.
2. Открыть `%LOCALAPPDATA%\Histshot\debug.log`.
3. Найти строки:
   - `Screen detected:` — проверить bounds и scaling.
   - `Overlay screenBounds=` — проверить, что окно создаётся для правильного монитора.
   - `Overlay after SetWindowPos:` — проверить `Position`, `Bounds`, `Width`, `Height`.
4. Открыть `%LOCALAPPDATA%\Histshot\capture_*.png` — изображения должны совпадать с экранами целиком.
5. Визуально проверить:
   - overlay появляется на каждом мониторе?
   - overlay покрывает весь монитор или занимает только часть?
   - изображение в overlay совпадает с тем, что под ним?

## Найденная причина (по debug.log)

Ключевые строки лога:

```
Overlay screenBounds=-2560, 778, 2560, 1600, scaling=1,75
Overlay after SetWindowPos: Position=-2560, 778, Bounds=0, 0, 1280, 800, Width=1280, Height=800
Overlay after SetWindowPos: Position=0, 0,     Bounds=0, 0, 1920, 1080, Width=1920, Height=1080
```

`Bounds` Avalonia хранит в DIP, окно физически растянуто `SetWindowPos`. Отсюда фактический
RenderScaling окна = физический размер / Bounds:

- **DISPLAY5** (primary, 3840×2160): `3840 / 1920 = 2.0` — совпадает с DPI монитора ⇒ всё верно.
- **DISPLAY1** (2560×1600, монитор 1.75): `2560 / 1280 = `**`2.0`**, а не 1.75.

То есть Avalonia создаёт overlay на primary-мониторе (scaling 2.0) и **не меняет RenderScaling**,
когда мы перетаскиваем HWND на DISPLAY1 через `SetWindowPos`: для окна, которое приложение двигает
само, `WM_DPICHANGED` Avalonia не отрабатывает.

Дальше весь код считал размеры контента от переданного `_scaling = 1.75`:
`_backgroundImage.Width = 2560 / 1.75 = 1462.86` DIP, тогда как видимый canvas — `1280` DIP
(окно рендерится при 2.0). Картинка шире окна → обрезается справа/снизу → выглядит как
«overlay занимает только часть монитора». На DISPLAY5 `2.0 == 2.0`, поэтому там проблемы нет.

## Решение (реализовано)

Причина состоит из двух частей:

1. **Ужатие окна по WM_DPICHANGED.** Когда `SetWindowPos` переносит overlay на монитор с другим
   DPI, Windows синхронно шлёт `WM_DPICHANGED`, и Avalonia ужимает окно до «suggested rect»:
   `2560×1600 × (1.75/2.0) = 2240×1400` физ. пикселей — меньше монитора. Один вызов `SetWindowPos`
   не даёт полноэкранного overlay. Лечится **двойным** `SetWindowPos` (приём из «Следующие шаги» п.1):
   первый перемещает и провоцирует смену DPI, второй (монитор уже тот же, смены DPI нет) фиксирует
   полный физический размер. Дополнительно размер переустанавливается на отложенном тике
   (`DispatcherPriority.Background`) — на случай асинхронного relayout.

2. **Несогласованный масштаб контента.** Все DIP↔пиксель вычисления переведены с переданного
   `_scaling` (DPI монитора) на фактический `this.RenderScaling` — масштаб, в котором Avalonia
   реально рисует окно (тот же приём, что уже работает в `EditorWindow.OnOpened`). Поскольку HWND
   принудительно растянут на весь монитор, `Width = физический_размер / RenderScaling` заполняет его
   точно, какой бы DPI Avalonia ни присвоила окну.

Изменения в `CaptureOverlayWindow`:

- `StartCapture()` → `ForcePhysicalBounds()` (×2) + `ApplyContentLayout()`, плюс повтор на отложенном тике.
- `ApplyContentLayout()`: `Width/Height` и фон считаются от `this.RenderScaling`.
- `CurrentScaling => this.RenderScaling` (используется в кропе выделения и `CaptureResult`).

Фон — это нативный захват (2560×1600), отрисованный 1:1 в device px, поэтому изображение совпадает с
экраном независимо от RenderScaling. Косметический нюанс: рамки/тулбар могут рисоваться в масштабе
монитора-владельца (если Avalonia не сменила DPI) — приемлемо.

## Следующие шаги

Если после текущих изменений проблема сохраняется, стоит попробовать:

1. **Использовать `MoveWindow` дважды**, как рекомендуется для WPF/Win32 при перемещении между мониторами с разным DPI:
   - первый вызов перемещает окно и триггерит `WM_DPICHANGED`;
   - второй вызов устанавливает окончательный размер.

2. **Использовать Avalonia Screens API** (`TopLevel.GetTopLevel(this)?.Screens`) для получения bounds/scaling вместо прямых WinAPI-вызовов.

3. **Создавать окно сразу на нужном мониторе**, передавая позицию в `SetWindowPos` в момент создания HWND, до первого layout pass.

4. **Отказаться от DIP-размеров** и управлять размером окна только через WinAPI в физических пикселях, чтобы исключить влияние Avalonia scaling.

5. **Рассмотреть Windows.Graphics.Capture API** вместо GDI `BitBlt`/`CreateDC` — он корректно работает с PerMonitorV2 и разными DPI.
